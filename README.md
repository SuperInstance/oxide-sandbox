# oxide-sandbox

> *A scientific instrument for exploring the gap between agent-generated intent and silicon execution.*

[![Rust](https://img.shields.io/badge/rust-2021-edition-blue.svg)](https://rust-lang.org)
[![License](https://img.shields.io/badge/license-Apache--2.0-blue.svg)](LICENSE)

## The Verification Gap

There is a moment in the modern machine-learning pipeline that ought to feel uncomfortable to everyone who has thought about it. An agent—a language model, a planning system, a reinforcement-learning controller—generates bytecode expressing some computational intent. That bytecode passes through a compiler stack. Somewhere in that chain, the agent's intent becomes silicon instructions that execute inside a GPU streaming multiprocessor at femtojoules per operation. No human wrote the bytecode. No human can read the PTX. The agent might have generated wrong intent. The compiler might have silently introduced a semantic shift. The resulting kernel runs at four hundred thousand operations per second and the outputs flow into downstream systems.

**How do you know it's right?**

This is the *verification gap*, and `oxide-sandbox` exists to narrow it. It is not a production compiler. It is a **controlled laboratory** for the Flux→MIR→PTX compile path, built so that every stage can be instrumented, inspected, and challenged. If the SuperInstance ecosystem is to realize its ambition—agents writing GPU kernels in a high-level, verification-friendly intermediate language called Flux, executing as PTX on NVIDIA GPUs—then someone must first prove that the translation preserves meaning. That someone is this crate.

## Background: Why This Path Matters

The [SuperInstance](https://github.com/SuperInstance/SuperInstance) project aspires to a world where agentic cognition is not bottlenecked by human-written kernels. In that world, a fleet of ten thousand agents negotiates, plans, and computes on distributed GPUs, loading capabilities dynamically from git constructs, synchronizing state via CRDTs, and expressing numerical reasoning through ternary arithmetic—the `{-1, 0, +1}` domain that compresses neural weights, accelerates search, and admits exact algebraic closure under Z₃.

The gap between Flux's semantic world and GPU machine code is wide. Type systems differ. Memory models differ. The correctness invariants that hold in Flux may be violated by aggressive GPU optimization. Ownership semantics in Rust's Mid-level IR (MIR) do not map cleanly to agent-generated bytecode. Warp divergence can serialize what was intended as parallel. A ternary addition that is mathematically closed in Z₃ may be compiled to saturating integer arithmetic that silently drifts.

`oxide-sandbox` asks: given a sequence of Flux bytecode operations, can we produce a synthetic MIR representation that is both optimizable and verifiable? Can we detect when ternary operations preserve their conservation laws through compilation? Can we measure the semantic cost of every transformation, so that when a production compiler is eventually built, its design is informed by data rather than hope?

## How It Works

The heart of the sandbox is the `ExperimentalCompiler`, a five-stage pipeline that treats every compilation as a reproducible experiment. Each stage records timing, output size, and success status, producing an `ExperimentResult` that can be aggregated, compared, and regressed against.

### The Five Stages

| Stage | Purpose | Output |
|---|---|---|
| **`parse_bytecode`** | Accept a slice of `FluxOp` instructions and validate structural well-formedness. | Parsed operation vector with provenance metadata. |
| **`type_inference`** | Walk operations and assign MIR types (`I32`, `F32`, `Ternary`, `GpuPtr`). | Type map keyed by register, with ternary density metrics. |
| **`mir_generation`** | Lower Flux operations into `SyntheticMir` blocks containing `MirOp`s. | MIR basic blocks, register types, and GPU metadata (block dimensions, shared memory, SM target). |
| **`ptx_generation`** | Emit PTX-like text from MIR, including version directives, register declarations, and barrier instructions. | Simulated PTX lines, register count, and barrier count. |
| **`conservation_verify`** | Check that ternary operations (`TADD`, `TMUL`) compose correctly in Z₃, ensuring every ternary-defined register is used consistently. | Conservation pass/fail verdict. |

### Flux Operations

Flux bytecode in the sandbox spans the full spectrum from scalar integer math to GPU-specific primitives:

- **Scalar arithmetic:** `MOVI`, `ADD`, `SUB`, `MUL`
- **Ternary arithmetic:** `TADD`, `TMUL` — the `{-1, 0, +1}` operations that form a commutative ring isomorphic to Z₃
- **GPU indexing:** `ThreadIdx` — access to `%tid.x`, `%tid.y`, `%tid.z`
- **Synchronization:** `SyncThreads` — maps to `bar.sync 0`
- **Memory:** `Load`, `Store`
- **Control flow:** `Branch`, `Halt`

### Synthetic MIR

The sandbox lowers Flux into `SyntheticMir`, an intermediate representation designed for inspection rather than execution. MIR captures:

- Basic blocks with labeled `MirOp` sequences
- Register type annotations (`I32`, `F32`, `Ternary`, `GpuPtr`)
- GPU metadata: block dimensions, shared memory budget, minimum SM version

This MIR is intentionally simple. It omits the borrow checker, the panic path, and the monomorphization complexity of real Rust MIR so that researchers can focus on the semantic mapping from Flux to GPU semantics without drowning in rustc internals.

### Conservation Verification

The most distinctive feature of the sandbox is its ternary conservation verifier. In Z₃ arithmetic, `tadd` and `tmul` are closed operations: the result of combining two ternary values is always another ternary value. The verifier checks that every ternary result is used in another ternary operation, stored, or returned—ensuring that no ternary value leaks into an integer context where it would be misinterpreted.

This is a simplified but principled check. It does not yet answer whether loading a ternary value from global memory and storing it back preserves conservation across address spaces. That is an open question, documented in the test suite, waiting for the next researcher.

## Experiment Results

For every compilation, the sandbox produces an `ExperimentResult` containing:

- `total_duration_us` — end-to-end compile time in microseconds
- `stages_passed` — how many of the five stages succeeded
- `ptx_generated` — whether PTX emission completed
- `ptx_size_bytes` — estimated PTX artifact size
- `register_count` — number of virtual registers allocated
- `shared_mem_bytes` — shared memory footprint
- `thread_coherence` — a heuristic score from 0.0 to 1.0 measuring how well threads stay in lockstep
- `conservation_verified` — whether Z₃ conservation held

Multiple experiments can be batched through a single `ExperimentalCompiler` and summarized via `ExperimentSummary`, yielding aggregate pass rates, average compile times, and conservation verification rates.

## Canonical Experiments

The test suite encodes three paradigmatic experiments that serve as both regression tests and teaching examples.

### Simple Integer Compile

```rust
let mut compiler = ExperimentalCompiler::new();
let ops = vec![
    FluxOp::MOVI { reg: 0, imm: 42 },
    FluxOp::MOVI { reg: 1, imm: 10 },
    FluxOp::ADD { rd: 2, rs1: 0, rs2: 1 },
    FluxOp::Halt,
];
let result = compiler.run_experiment("simple_add", &ops, FluxSource::HumanWritten);
assert!(result.ptx_generated);
assert_eq!(result.stages_passed, 5);
```

Validates that basic integer code parses, types, lowers, emits PTX, and passes conservation trivially.

### Ternary Compile

```rust
let ops = vec![
    FluxOp::MOVI { reg: 0, imm: 1 },
    FluxOp::MOVI { reg: 1, imm: -1 },
    FluxOp::TADD { rd: 2, ra: 0, rb: 1 },
    FluxOp::TMUL { rd: 3, ra: 2, rb: 0 },
    FluxOp::Halt,
];
let result = compiler.run_experiment("ternary_ops", &ops,
    FluxSource::AgentGenerated {
        model: "gpt-4".into(),
        intent: "compute ternary product".into(),
    });
assert!(result.conservation_verified);
```

Validates that `{-1, 0, +1}` arithmetic is recognized as `MirType::Ternary` and that the conservation verifier confirms closure in Z₃.

### GPU Kernel Compile

```rust
let ops = vec![
    FluxOp::ThreadIdx { dest: 0, dim: 0 },
    FluxOp::MOVI { reg: 1, imm: 100 },
    FluxOp::TADD { rd: 2, ra: 0, rb: 1 },
    FluxOp::SyncThreads,
    FluxOp::Halt,
];
let result = compiler.run_experiment("gpu_ternary", &ops,
    FluxSource::ConstructLoaded {
        repo: "SuperInstance/ternary-attention".into(),
        version: "v2.0.0".into(),
    });
```

Validates that GPU-specific operations produce correct MIR and PTX barrier instructions, and that constructs loaded from repositories carry their provenance through the pipeline.

## Applications

- **Compiler research:** Test new lowering strategies before they enter the production `cuda-oxide` path. The sandbox's instrumented stages make it easy to A/B different translation schemes and measure their semantic and performance impact.
- **Agent-generated code validation:** Verify that LLM-generated Flux kernels produce well-formed MIR before deployment. When an agent emits bytecode, the sandbox can reject it at compile time if conservation laws are violated or if the resulting PTX exceeds resource bounds.
- **Curriculum tool:** Teach GPU compilation stages with a minimal, instrumented pipeline. Students can read the entire compiler in a single sitting, modify a stage, and immediately see the effect on emitted PTX.
- **Construct integration:** Validate constructs from the SuperInstance ecosystem before they are deployed to live GPUs. A construct that fails conservation verification in the sandbox should never reach the fleet.
- **Regression detection:** Run the sandbox on every commit to catch semantic drift in the Flux language. If a change to the Flux spec causes previously-passing experiments to fail conservation, the regression is caught before it reaches production.
- **Physics-based type systems:** The conservation verifier is a prototype of something richer: a type system where `Ternary` is not merely a tag but a contract backed by Noether's theorem. Future work could integrate energy conservation, momentum conservation, and Liouville phase-space volume checks as compile-time passes.

## Open Questions

The sandbox is honest about what it does not yet know:

1. **Conservation under memory operations:** The current verifier only checks ternary register use. Does loading a ternary value from global memory and storing it back preserve conservation across address spaces?
2. **Thread divergence and coherence:** How should `thread_coherence` be defined in the presence of divergent branches? The current heuristic assumes perfect coherence; real GPU kernels diverge.
3. **Optimization soundness:** If we add MIR-level constant folding, does it preserve ternary conservation for all Z₃ expressions?
4. **PTX emission fidelity:** How closely must simulated PTX match real PTX for the sandbox to be a valid proxy for the production pipeline?

These questions are not bugs. They are the research agenda.

## Quick Start

```rust
use oxide_sandbox::{ExperimentalCompiler, FluxOp, FluxSource};

let mut compiler = ExperimentalCompiler::new();
let ops = vec![
    FluxOp::MOVI { reg: 0, imm: 1 },
    FluxOp::MOVI { reg: 1, imm: -1 },
    FluxOp::TADD { rd: 2, ra: 0, rb: 1 },
    FluxOp::Halt,
];

let result = compiler.run_experiment(
    "ternary_test",
    &ops,
    FluxSource::AgentGenerated {
        model: "gpt-4".into(),
        intent: "compute ternary sum".into(),
    },
);

assert!(result.ptx_generated);
assert!(result.conservation_verified);
println!(
    "Compiled in {} us across {} stages",
    result.total_duration_us, result.stages_passed
);
```

Run the test suite:

```bash
cargo test
```
